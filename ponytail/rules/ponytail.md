# ponytail — 防过度工程规则（always-on）

灵感来自 [ponytail](https://github.com/DietrichGebert/ponytail)：像房间里最懒的资深开发者一样思考。**最好的代码是你从未写过的代码。**

写任何代码前，按下面的「判断阶梯」逐级自问，**停在第一个成立的台阶**，不要继续往下堆复杂度：

```
1. 这东西需要存在吗？      → 不需要：直接跳过（YAGNI，别造没人要的功能/抽象）
2. 标准库能做吗？          → 能：用标准库
3. 平台原生功能能做吗？    → 能：用原生（如 <input type="date"> 而非引日期库）
4. 已安装的依赖能做吗？    → 能：复用现有依赖，别加新包
5. 一行能搞定吗？          → 能：就一行
6. 实在不行：写「能用的最小实现」，不提前抽象、不加未来才需要的可配置项
```

## 安全红线（绝不为了"少写"而牺牲）

以下内容**永远不省**，ponytail 只砍冗余，不砍质量：
- 输入校验、边界处理
- 错误处理 / 异常路径
- 安全（鉴权、注入防护、敏感信息处理）
- 可访问性（a11y）

## 反面信号（出现就该收手）

- 为一个简单需求安装第三方库（先想原生/标准库）
- 写包装层 / 工厂 / 抽象，但当前只有一个实现
- 加"以后可能用得到"的参数、配置、扩展点
- 把一行能写的拆成多文件多类

## 强度档位（可被 /ponytail 命令切换，默认 full）

- `lite`：仅在明显过度时提醒
- `full`：默认，每次写代码前走判断阶梯
- `ultra`：激进精简，优先质疑"是否需要写代码"
- `off`：临时关闭本规则

> 延后处理的精简点用 `// ponytail: <说明>` 标记，便于之后用 /ponytail-debt 汇总。
